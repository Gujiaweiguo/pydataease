## Context

当前 FastAPI 后端通过 `app.main` 聚合各业务 router，并统一套用鉴权中间件与 `ResultMessage` 响应包装。同步引擎模块在前端已经存在调用入口，但后端缺失 `/sync/**` 路由，导致前端在数据源、任务、日志、摘要四类页面上的请求全部失败。

这次改动的目标不是接入真实同步能力，而是在不引入 service/repository/数据库复杂度的前提下，先补上稳定 API 外壳，让前端契约和权限链路可验证。

## Goals / Non-Goals

**Goals:**
- 在 FastAPI 后端新增单一 `sync` router，覆盖前端依赖的 30 个同步相关端点。
- 保持所有端点经过现有鉴权和响应包装中间件，避免新增特例逻辑。
- 使用固定 stub 负载返回空数据结构，让前端不再收到 404，并可继续开发页面交互。
- 为路径注册和鉴权要求补充测试，确保后续真实实现替换时不会意外丢失接口。

**Non-Goals:**
- 不实现 SeaTunnel、调度器、日志流或真实数据同步能力。
- 不新增 schema/service/repository/model，也不落库。
- 不调整前端调用方式，不引入新的 API 前缀或版本。

## Decisions

1. **使用单一 router 文件集中承载所有 sync stub**  
   - 选择：新增 `app/routers/sync.py`，在一个文件内定义全部 30 个端点。  
   - 原因：当前需求纯粹是补齐契约，占位逻辑极轻；集中定义最易审查，也便于未来整体替换为真实模块。  
   - 备选方案：按 datasource/task/log/summary 再拆分多个 router 文件。暂不采用，因为会增加文件和注册复杂度，但不会带来真实分层收益。

2. **复用全局鉴权链路，而不是在 router 内自定义鉴权**  
   - 选择：每个端点显式依赖 `get_current_user`。  
   - 原因：sync 模块应与现有业务路由保持一致，确保未登录访问返回 401，并与后续真实实现的权限边界对齐。  
   - 备选方案：仅依赖全局中间件或暂时放开匿名访问。暂不采用，因为会削弱测试可见性，并偏离前端对受保护接口的预期。

3. **直接返回字典/列表字面量，占位响应尽量简单**  
   - 选择：stub 端点直接返回 `dict`/`list`，由现有响应包装中间件包装成统一 `{code,data,msg}`。  
   - 原因：避免为临时空实现创建大量一次性 schema；占位结构与用户给定的契约一一对应。  
   - 备选方案：为每个响应定义 Pydantic schema。暂不采用，因为当前没有真实业务字段，收益低于维护成本。

4. **测试聚焦路径注册和代表性鉴权行为**  
   - 选择：新增 `tests/test_sync_routes.py`，检查 30 个路由均已注册，并对若干 GET/POST 接口验证无 token 时返回 401、有 token 时返回预期占位数据。  
   - 原因：这类改动的风险主要是路径遗漏、方法不匹配、router 未注册或误绕过鉴权，而非业务计算错误。  
   - 备选方案：为 30 个端点逐个发请求做全覆盖。暂不采用，因为重复高、信噪比低。

## Risks / Trade-offs

- **[风险] stub 返回结构过于宽松，未来真实实现替换时可能出现字段漂移** → **缓解**：在 spec 中固定每类端点当前应返回的最小结构，并用测试锁定代表性响应。
- **[风险] 单文件集中 30 个端点会偏长** → **缓解**：按 datasource/task/log/summary 分段组织，并保持函数名清晰，未来真实实现时再按模块拆分。
- **[权衡] 暂不定义响应 schema 会减少类型约束** → **缓解**：当前仅用于占位；后续真实实现再引入稳定 schema，避免现在为临时空结构过度设计。
